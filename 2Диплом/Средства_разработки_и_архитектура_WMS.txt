Для разработки веб-приложения WMS (система управления складом автозапчастей) используется фреймворк Django версии 5.2.10 на языке Python 3.12.4.
Данный стек выбран для проекта по следующим причинам:
Встроенные механизмы безопасности Django (CSRF, защита шаблонов от XSS, безопасная работа с БД через ORM);
Готовая система аутентификации и авторизации пользователей;
Поддержка архитектуры MVT, что упрощает разделение логики, данных и интерфейса;
Наличие административного интерфейса и развитой экосистемы;
Удобная интеграция с PostgreSQL и Celery;
Возможность масштабирования и развертывания в разных средах.

Фреймворк Django в проекте реализует принцип batteries included: в системе используются встроенные компоненты для форм, сессий, сообщений, маршрутизации, шаблонов и проверки прав доступа по ролям пользователей.

Для хранения данных используется PostgreSQL (подключение задается через DATABASE_URL в settings.py, с использованием dj_database_url).
PostgreSQL выбран по следующим причинам:
Поддержка транзакционной модели и целостности данных;
Надежная работа при интенсивных CRUD-операциях склада;
Поддержка сложных связей между сущностями (товары, приемки, заказы, задачи, отчеты);
Высокая производительность при работе с индексируемыми справочниками и операционными таблицами;
Широкая поддержка в Django-проектах.

В окружении проекта используются драйверы psycopg2 и psycopg для взаимодействия Django с PostgreSQL.

API-слой в проекте реализован как отдельный модуль (api, api/v1) на базе Django views с JSON-ответами и собственной утилитной обвязкой (json_ok/json_error, paginate, parse_json).
Аутентификация API реализована через Bearer-токены (модель ApiToken) и проверку роли интеграционного пользователя.
Дополнительно реализована встроенная документация API через OpenAPI/Swagger/ReDoc страницы (templates/api/v1/swagger.html, templates/api/v1/redoc.html).

Важно: в окружении установлен Django REST Framework 3.14.0, однако в текущей версии проекта основной API-контур реализован собственными Django views, без обязательной зависимости на DRF ViewSet-архитектуру.

Для асинхронных задач используется Celery (версия 5.6.2) в связке с Redis.
В проекте Redis выполняет две роли:
Кэш приложения (при наличии CACHE_URL);
Брокер и result-backend для Celery-задач (CELERY_BROKER_URL/CELERY_RESULT_BACKEND).

Фоновая обработка применяется, в частности, для интеграции с внешним сервисом EFP.
В модуле efp.queue реализованы:
Постановка задач в очередь;
Проверка доступности Redis/Celery;
Контроль статусов фоновых операций;
Fallback-режим в локальный поток при недоступности очереди (по флагу EFP_ALLOW_THREAD_FALLBACK).

Внешняя интеграция с EFP построена в efp.services с использованием библиотек:
requests для HTTP-запросов;
beautifulsoup4 для парсинга HTML-ответов;
кэширования результатов поиска для снижения нагрузки.

Для формирования документов PDF используется библиотека reportlab.
В проекте она применяется для генерации:
руководства пользователя (core/views.py);
печатных документов по приемке (receiving/views.py).

Frontend-часть проекта построена как серверно-рендеримый интерфейс на Django Template Language (DTL):
HTML-шаблоны в каталоге templates/;
кастомные стили в static/css/;
кастомные JavaScript-модули в static/js/.

В проекте не используется Bootstrap как базовый UI-фреймворк: интерфейс построен на собственных CSS-стилях и JS-логике.
Для 3D-визуализации склада применяется Three.js (templates/warehouse_3d/view.html).
Для API-документации подключается ReDoc.

Архитектурно приложение организовано по принципу MVT с прикладным сервисным слоем:
Model: модели предметной области в apps (catalog, receiving, inventory, picking, tasks, reports и др.);
View: веб-представления и API endpoint-ы;
Template: HTML-шаблоны пользовательского интерфейса;
Service layer: бизнес-операции вынесены в services.py (например, ReceivingService, InventoryService, OrderService, TaskService).

Ключевые функциональные подсистемы проекта:
Каталог и справочники (бренды, категории, склады, зоны, ячейки);
Приемка товаров и размещение по местам хранения;
Учет остатков и инвентаризация;
Заказы и операции подбора;
Контур задач и мониторинг исполнения;
Аналитические отчеты;
3D-представление склада;
Внешняя интеграция с EFP.

Система построена как единое веб-приложение с общей аутентификацией, ролевой моделью доступа и централизованной БД, что упрощает сопровождение и обеспечивает согласованность бизнес-процессов.

Для управления исходным кодом используется Git.
Для управления зависимостями применяется pip и виртуальное окружение Python (.venv).

Практика развертывания для production может включать:
WSGI/ASGI-сервер для запуска Django;
Reverse proxy (например, Nginx) для маршрутизации и отдачи статических файлов;
Выделенные процессы Celery worker и Redis;
PostgreSQL как отдельный сервис БД.

Текущий технологический стек проекта выбран с учетом требований к надежности складских операций, безопасности, расширяемости бизнес-логики и удобству дальнейшей поддержки системы.
