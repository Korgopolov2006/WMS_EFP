В ходе написания программы были использованы следующие методы разработки программного обеспечения и архитектурные паттерны.

  Методология ООП (Объектно-ориентированное программирование).
Объектно-ориентированный подход является базой всей серверной части проекта. Программная система представлена набором взаимосвязанных классов предметной области, каждый из которых инкапсулирует структуру данных и поведение.
В проекте реализовано через:
  Классы моделей, наследующиеся от django.db.models.Model, например User, Warehouse, Product, Receiving, Stock, Order, PickingTask, Task, WarehouseLayout, StorageObject;
  Использование абстрактных базовых моделей (например, TimeStampedModel) для повторного использования общих полей created_at и updated_at;
  Классы сервисного слоя (ReceivingService, OrderService, PickingService, TaskService и другие), инкапсулирующие бизнес-операции;
  Логическое разделение ответственности между моделями, представлениями и сервисами в соответствии с SRP;
  Использование перечислений и констант классов (Roles, статусы документов и задач) для типобезопасного управления состояниями.

  Паттерн MVT (Model-View-Template).
Архитектурный каркас Django реализован в полной мере: модель описывает данные и ограничения, представление обрабатывает запрос, шаблон отображает результат. Это обеспечивает предсказуемую структуру проекта и удобную поддержку.
В проекте реализовано через:
  Model: файлы models.py в модулях accounts, catalog, receiving, inventory, picking, tasks, reports, warehouse_3d, api;
  View: функциональные представления в views.py, включая ролевой доступ, фильтрацию, обработку форм и бизнес-действий;
  Template: HTML-шаблоны в каталоге templates с использованием Django Template Language, разбивкой на экраны и частичные шаблоны;
  Отдельные маршруты URL в каждом модуле для декомпозиции навигации и API-точек.

  RESTful API архитектура (интеграционный контур).
В проекте реализован API в стиле REST для внешних интеграций и автоматизированных сценариев. Взаимодействие строится на HTTP-методах, JSON-представлении данных и stateless-принципе.
В проекте реализовано через:
  Набор endpoint-ов в api/v1/urls.py для справочников, товаров, приемки, остатков, инвентаризаций, заказов, задач, отбора, отчетов и 3D-данных;
  Использование методов GET, POST, PUT/PATCH, DELETE для операций чтения и изменения сущностей;
  Единообразные JSON-ответы и JSON-ошибки через утилиты api/utils.py (json_ok, json_error);
  Валидацию входного JSON на уровне API-слоя (parse_json, проверки обязательных полей, проверка числовых значений);
  Bearer-аутентификацию через интеграционные токены (ApiToken, authenticate_integration в api/auth.py);
  Документирование API через OpenAPI + Swagger/ReDoc (api/v1/openapi.py, api/v1/docs.py).

  Паттерн ORM (Object-Relational Mapping).
Django ORM используется как основной механизм доступа к PostgreSQL. Это позволяет работать с данными на уровне объектов Python, обеспечивая переносимость, безопасность и контроль ограничений.
В проекте реализовано через:
  Декларативные модели и связи (ForeignKey, OneToOne, ManyToMany);
  Миграции schema-first через встроенный механизм migrations;
  Ограничения целостности на уровне модели: unique, UniqueConstraint, индексы, choices;
  Сложные выборки через QuerySet API (filter, annotate, aggregate, order_by, distinct);
  Защиту от SQL-инъекций за счет параметризованного ORM-доступа;
  Оптимизацию SQL-запросов через select_related и prefetch_related.

  Service Layer (Сервисный слой).
Ключевая бизнес-логика вынесена из представлений в отдельные сервисные классы. Это повышает тестируемость, упрощает сопровождение и уменьшает связанность между UI/API и доменной логикой.
В проекте реализовано через:
  receiving/services.py: валидация и завершение приемки, создание/обновление остатков, завершение связанных задач;
  inventory/services.py: запуск и завершение инвентаризаций, расчет расхождений, синхронизация фактических данных;
  picking/services.py: подтверждение заказа, создание задач отбора, фиксация подбора, отгрузка и списание резервов;
  tasks/services.py: создание и маршрутизация задач по ролям, назначение, завершение, контроль статусов;
  reports/services.py: расчеты аналитики (ABC/XYZ, dead stock, прогноз, ошибки отбора, эффективность персонала);
  catalog/services.py: логика изменений каталога, кросс-ссылок и связанных процессов.

  Транзакционный паттерн (Transaction Script / Unit of Work на уровне ORM).
Критические операции выполняются атомарно, чтобы сохранить консистентность данных между документами, остатками и задачами.
В проекте реализовано через:
  Использование @transaction.atomic в сервисных операциях приемки, инвентаризации, подтверждения/отгрузки заказов и сценариях наполнения;
  Единые транзакционные блоки для обновления нескольких связанных сущностей;
  Контроль ошибок на ранних этапах операции до коммита;
  Защиту от частичного применения бизнес-операции при сбое.

  Паттерн декораторов.
Декораторы используются для внедрения сквозной функциональности без дублирования кода в каждой view-функции.
В проекте реализовано через:
  @login_required для обязательной аутентификации пользователя;
  Кастомный @role_required(*roles) для ролевого контроля доступа на уровне бизнес-разделов;
  Встроенные механизмы Django для CSRF-защиты и middleware-обработки;
  Единообразные правила доступа в интерфейсных и API-сценариях.

  Role-Based Access Control (RBAC).
Доступ к функциям системы управляется ролями пользователей и привязкой к филиалам/складам. Это снижает риск несанкционированных действий и разграничивает зоны ответственности сотрудников.
В проекте реализовано через:
  Ролевую модель Roles: ADMIN, STOREKEEPER, SMALL_PARTS_PICKER, LOADER, SALES_MANAGER, ANALYST, INTEGRATION;
  Проверки роли в декораторе role_required;
  Ограничение видимости и действий пользователя в представлениях и сервисах;
  Модель WarehouseAccess для управления доступом пользователей к складам;
  Специальную роль INTEGRATION для API-токенов и внешнего обмена.

  Паттерн DRY (Don’t Repeat Yourself).
Принцип повторного использования кода применен для снижения дублирования и упрощения поддержки.
В проекте реализовано через:
  Базовые модели с общими полями времени;
  Общие функции пагинации в модулях web-интерфейса и API;
  Унифицированные функции JSON-ответов и обработки ошибок;
  Централизацию бизнес-логики в сервисах вместо копирования ее в view;
  Переиспользование шаблонов и partial-компонентов интерфейса.

  Принцип разделения ответственности (Separation of Concerns).
Проект разделен по функциональным областям, а внутри каждой области — по техническим слоям. Это улучшает читаемость и снижает стоимость изменений.
В проекте реализовано через:
  Выделение отдельных приложений: catalog, receiving, inventory, picking, tasks, reports, api, efp, warehouse_3d, accounts;
  Разделение веб-представлений и API-представлений;
  Отделение представления данных (templates) от бизнес-операций (services) и хранения (models);
  Изоляцию интеграционных сценариев (efp, api) от внутреннего пользовательского интерфейса.

  Паттерн Pagination (Пагинация).
Пагинация применяется для уменьшения нагрузки на БД и улучшения UX при работе с большими выборками.
В проекте реализовано через:
  Django Paginator в веб-интерфейсе (каталог, приемка, остатки, задачи, отчеты, заказы и другие списки);
  limit/offset-пагинацию в API через общую функцию paginate;
  Ограничение максимального размера выдачи для предотвращения чрезмерных ответов;
  Возврат метаданных пагинации (limit, offset, total) в API.

  Middleware-подход (сквозная обработка запросов и ответов).
Глобальные аспекты безопасности и состояния обрабатываются на уровне middleware-цепочки Django.
В проекте реализовано через:
  SecurityMiddleware для базовых механизмов защиты HTTP-уровня;
  SessionMiddleware для сессионного контекста пользователя;
  CommonMiddleware для стандартной обработки запросов;
  CsrfViewMiddleware для защиты от CSRF-атак;
  AuthenticationMiddleware для привязки пользователя к запросу;
  MessageMiddleware для системных уведомлений;
  XFrameOptionsMiddleware для защиты от clickjacking.

  Асинхронная архитектура фоновых задач (Celery + Redis).
Для задач, не требующих мгновенного завершения в HTTP-запросе, используется отдельный асинхронный контур. Это снижает время отклика интерфейса и повышает отказоустойчивость интеграций.
В проекте реализовано через:
  Инициализацию Celery-приложения в wms/celery.py;
  Настройки брокера и result backend через Redis в settings.py;
  Фоновую задачу efp.search_oem для интеграционных запросов по запчастям;
  Хранение состояния выполнения задач в кэше с TTL и контролем «устаревших» задач;
  Проверку доступности Redis и Celery worker перед постановкой задачи;
  Опциональный fallback в локальный поток при недоступности очереди.

  Паттерн кэширования.
Кэширование применяется для снижения количества повторных вычислений и ускорения интеграционного контура.
В проекте реализовано через:
  Использование Django CACHES (RedisCache или LocMemCache в зависимости от окружения);
  Кэширование health-check результатов для Redis/Celery в EFP-контуре;
  Кэширование промежуточных статусов фоновых заданий;
  Управление временем жизни записей (TTL) для предотвращения накопления устаревших данных.

  Паттерн документирования API по спецификации OpenAPI.
Для интеграторов реализована машиночитаемая спецификация API и web-представление документации.
В проекте реализовано через:
  Генерацию спецификации OpenAPI в api/v1/openapi.py;
  Endpoint /api/v1/openapi.json с актуальной схемой;
  Страницы /api/v1/docs/ (Swagger UI) и /api/v1/redoc/ (ReDoc);
  Единые описания параметров запросов и моделей ответа.

  Принципы defensive programming (защитное программирование).
Для устойчивой работы бизнес-процессов применяются проверки допустимости состояний и входных данных до изменения БД.
В проекте реализовано через:
  Явную валидацию входных параметров в API и сервисах;
  Контроль допустимых переходов статусов документов и задач;
  Проверку количеств, доступности остатков и прав пользователя до выполнения операций;
  Возврат диагностических сообщений об ошибках с понятной причиной отказа.

  Pattern of explicit state machine (явное управление состояниями объектов).
Документы и задачи проходят фиксированные этапы жизненного цикла, что делает поведение системы предсказуемым и проверяемым.
В проекте реализовано через:
  Перечисления статусов ReceivingStatus, InventoryStatus, OrderStatus, PickingTaskStatus, TaskStatus;
  Бизнес-правила переходов в сервисах (например, нельзя отгрузить неподобранный заказ);
  Фиксацию временных отметок этапов started_at, completed_at, confirmed_at, shipped_at;
  Согласованное закрытие связанных задач при завершении бизнес-операции.

  Management Commands как эксплуатационный паттерн.
Критичные административные и подготовительные операции вынесены в отдельные команды, что стандартизирует эксплуатацию системы.
В проекте реализовано через:
  Команды наполнения демо/тестовыми данными для различных модулей и ролей;
  Команду rebuild_schema_sql для генерации актуального DDL из PostgreSQL;
  Команду create_integration_token для выпуска и ротации API-токенов;
  Команды диагностики и подготовки отчетных/мониторинговых данных.

  SOLID-принципы (практическое применение).
В проекте SOLID применяется прагматично, без избыточного усложнения архитектуры.
В проекте реализовано через:
  Single Responsibility Principle: отдельные приложения и сервисы отвечают за конкретные бизнес-области;
  Open/Closed Principle: добавление новых сценариев обычно происходит через новые сервисы/обработчики без переписывания ядра;
  Liskov Substitution Principle: использование полиморфизма и базовых контрактов Django-моделей/классов;
  Interface Segregation Principle: разделение web- и API-контуров, а также специализированных endpoint-ов;
  Dependency Inversion Principle: опора на абстракции Django (ORM, middleware, auth, cache), а не на привязку к низкоуровневым механизмам.

  Отдельное замечание по Django Signals.
В текущей версии проекта ключевая бизнес-логика преднамеренно не вынесена в сигналы post_save/pre_save. Вместо этого она сконцентрирована в явных сервисных сценариях. Такой подход выбран для прозрачности, управляемости транзакций и снижения неявных побочных эффектов.
Практический эффект выбранного подхода:
  Проще трассировать последовательность бизнес-действий;
  Ниже риск скрытых каскадных изменений данных;
  Удобнее тестировать операции как целостные use-case сценарии;
  Проще контролировать точки входа для интеграций и аудита.

Таким образом, архитектура проекта сочетает классический подход Django (MVT + ORM + middleware) с сервисной декомпозицией бизнес-логики, ролевой моделью доступа, интеграционным REST API, асинхронной обработкой задач и эксплуатационными инструментами сопровождения. Это обеспечивает масштабируемость, предсказуемость поведения системы и удобство дальнейшего развития.
